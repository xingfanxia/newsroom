## 1. Voice journey (this is the most important part of this handoff)

The spec called for "khazix-pure 公众号 voice" but operator feedback after each iteration pushed us toward a different register. **Final voice = 虎嗅有意思周报** (渣渣郡 / 木子童 / 黄瓜汽水 register).

| Iteration | PR | Voice register | What was wrong | What worked |
|---|---|---|---|---|
| **v1** | #21 | khazix-pure (公众号狠口语化, "活人感至上", 一句话独立成段) | Voice "太随意" — operator said this is for daily news, not 公众号 long-form. Closers like 不是哥们/愚钝如我/卧槽 felt out of place for a news brief. Single through-narrative, no sectioning. | Got the framework live: schema, writer, RSS, MCP, skill, page UI. |
| **v2** | #25 | Stratechery / Matt Levine / 阮一峰 (产业分析师 register) | "Reads weird" — too analyst-y. "我的判断是" overused. Markdown structure required but rendering CSS was tight. | Picked up: ## subheadings + ### sub-headings + > blockquotes + **bold** allowed/encouraged. CSS framework added. |
| **v3** | #26 | **虎嗅有意思周报** (light + observational + dry humor + sectional) | (current — operator approved "好多了") | 6-10 ## sections, each a 200-1000字 mini-article with hook → core facts → context → optional > quote → observation → optional dry one-liner closer. Section titles are virhuxiu-style hooks ("Google 四百亿不是一张支票，是云上的订座费"). |

### Final voice spec (canonical)

Source of truth: **`lib/llm/prompts/daily-column.md`** (loaded by cron at runtime via `loadDailyColumnPrompt()`).
Mirror: `~/.claude/skills/ax-radar-daily-column/references/voice-spec.md` (copy-forward when source updates).

Key directives:
- **Reference**: 虎嗅有意思周报, NOT Stratechery/Matt Levine/阮一峰/卡兹克
- **Per-section structure**: hook → core facts (with `**bold**` numbers/names) → context → optional `>` blockquote → observation → optional dry one-liner closer
- **Voice register**: 值得留意 / 老实说 / 在我看来 OK; "我的判断是" capped at 1-2/article; 不是哥们/卧槽/愚钝如我/太离谱了 BANNED (公众号 register)
- **Dry closers** (optional, encouraged): "这一笔很贵, 但很 Google", "好吧", "至于这能不能行, 几个月后再看流程怎么改"
- **Length**: 6-10 sections × 200-1000字 = 5000-8000字 narrative + 100-200字 opening blurb

### What about the "khazix-pure" framework?

The L1-L2 self-check from the original khazix-pure spec was rebased:
- **L1 banned phrases** kept (说白了 / 综上所述 / 本质上 / 这给我们的启示是 — corporate AI-slop is universally bad)
- **L2 punctuation** rules (no colons / no em-dashes / no double quotes) **dropped entirely** — professional voice needs them for proper structure
- **First/其次/最后** removed from banned list (too restrictive for newsletter sectional flow)

## 6. Carried gotchas / lessons (do not lose)

1. **Structured-output mode is sensitive to schema constraint strictness.** Setting `narrative_md.min(4000)` caused JSON parse failures in `generateStructured` (Azure was retrying / model dropped non-JSON preamble). Dropped to `min(2500)` — works reliably. Verbose `.describe()` strings on schema fields also seemed to confuse the model into adding preambles. Keep schema descriptions terse for Azure structured output.

2. **Voice iteration is high-leverage and cheap.** Each prompt rewrite + smoke + manual eyeball cycle was ~5-10 min. Three iterations from khazix → Stratechery → 虎嗅 was the right tempo. Don't try to nail voice on the first try — ship a framework that supports easy iteration, then iterate against operator feedback.

3. **CSS spacing matters as much as voice for "professional feel."** v3 (Stratechery voice) shipped with the same tight Prose CSS as v1, and the operator perceived it as "formatting挺差的" even though the voice was much better. The article-grade CSS in v4 (18px paragraph margin, 56px before ##, blockquote panels) fixed perceived quality without touching the prompt.

4. **Linkify item refs at render time, not in the prompt.** Telling the model to emit `[#1234](/zh/items/1234)` directly didn't work consistently. Instead, the prompt emits `[#1234]` and the renderer rewrites to markdown link via regex before passing to react-markdown. Reliable, single-place control.

5. **Match the site's existing CSS shell, don't rebuild.** Custom Tailwind on the daily pages looked wrong against the rest of the site's mono+terminal aesthetic. Wrapping in `ViewShell` + using `.item` / `.feed` / `.page-head` matches every other tab automatically and saves design work.

6. **`force=true` on regen is the right default** — operator wants overwrites when they regen. The schema's unique key `(kind, locale, periodStart)` enforces 1 row per cron-hour-window; force lets that row update without an exists-check skip.

7. **Backfill needs all 7 day's selection windows in advance** — the script computes `<date>T05:00:00Z` for each day, then `selectDailyColumnPool` snaps the window-end to that hour. If items aren't enriched yet at that historical timestamp, pool can be empty and the day skips. Wasn't an issue today (8800+ items already enriched), but worth flagging.

## 8. What surprised us

- **Stratechery register felt off in Chinese.** I assumed "professional newsletter" = Stratechery-style analysis, but in Chinese that register reads as 学究 / 报告体. 虎嗅有意思周报 is the right Chinese-language analog: same intellectual seriousness, much lighter touch.
- **Cluster-dedup at selection time wasn't enough** — even after deduping by `cluster_id`, there were still near-dupe items (e.g., 5 different clusters all about Google→Anthropic). Stage A recall is the upstream issue. Mitigation: prompt instructs the writer to merge near-dupes in narrative.
- **`maxTokens: 12000` was tight for 6000+字 narrative.** Reasoning tokens (high effort) ate ~4-6K, output ~6K, total ~10-12K. Bumped to 30000 for headroom. Not a cost concern at 1 cron tick/day.
- **`bun:test` vs `vitest`** — newsroom uses `bun:test`. I wrote tests in vitest initially out of habit; bun emitted clear "Cannot find module 'vitest'" errors. Easy fix but worth flagging in the codebase's testing conventions doc.
- **Backfill ran fine in parallel with prod cron.** Operator concerns about LLM rate-limit collisions didn't materialize — Azure GPT-5.5 quota handles 8-9 concurrent calls without throttling. Good signal for future bulk regens.
