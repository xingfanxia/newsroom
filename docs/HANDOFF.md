## Architecture decisions locked in this session

### Inference: **Azure-only** (100% of LLM + embedding calls go to Azure)
- Free credits on the user's Azure account were the deciding factor
- Unified via Vercel AI SDK v6 + `@ai-sdk/openai` (for pro) + `@ai-sdk/azure` (for standard)
- Anthropic + Gemini providers still wired for optional fallback but not called by M2 workers
- See `project_azure_resources.md` memory for the two Azure resources and their endpoint shapes

### Per-task model profiles (in `lib/llm/profiles`)
| Task | Provider | Deployment | Reasoning | Notes |
|---|---|---|---|---|
| `enrich` (summary+tags) | `azure-openai` | gpt-5.4-standard | `low` | deterministic, cheap |
| `score` (importance+tier) | `azure-openai` | gpt-5.4-standard | `high` | NOT pro — pro's medium floor was ~3x slower than standard's high at ~95% quality |
| `agent` (M4 policy rewrite) | `azure-openai-pro` | gpt-5.4-pro-standard | `xhigh` | low-volume, reasoning-critical |
| embeddings | `azure-openai` | text-embedding-3-large | n/a | native 3072 dims |

**Reasoning effort constraints per model** (discovered empirically):
- gpt-5.4-standard: supports `minimal | low | medium | high`
- gpt-5.4-pro: supports `medium | high | xhigh` (rejects low/minimal)
- Claude Opus 4.7, Gemini 3.1 Pro, GPT-5: reject `temperature` parameter
- Our `lib/llm` code omits temperature unless caller explicitly sets it

### Database: **Supabase Postgres** (not Neon — user cited cold-start)
- Pooled URL for runtime (`postgres-js` + `prepare: false` + `max: 1`)
- Non-pooled URL for migrations (`drizzle-kit`)
- pgvector 0.8 enabled; `halfvec(3072)` column for embeddings (not `vector(1536)`)
  - rationale: same storage, better quality, native text-embedding-3-large width, fits HNSW's 4000-dim limit

### Deployment
- `vercel.json` has 6 cron schedules (fetch-hourly/daily/weekly, normalize, enrich, cluster)
- Fluid Compute with `maxDuration = 800` per route
- `CRON_SECRET` auto-generated by Vercel (not in our env — don't try to set it)
- Vercel ↔ GitHub linked; `git push origin main` auto-deploys production

## Deferred items from M2 ultra-review

All three reviewers (code + security) gave CRITICAL/HIGH findings; everything blocking was fixed. Deferred with documented rationale:

- **[H1] Staged persistence in enrich**: if stage 3 (score) fails after stages 1+2, we re-pay for all three on retry. Tokens are "free" on Azure credits so low priority. Would need `embedded_at` timestamp column + split UPDATEs.
- **[H2] Cross-cron row claim**: pLimit(4) within one process is safe, but two overlapping cron invocations both query 50 pending, both pay for LLMs, second UPDATE is no-op. At /15min cadence + 5s/item × 50 items = 4min, overlap is rare.
- **[H3] Policy as system prompt (~2500 tokens × 50 calls = 125K/batch)**: on Azure free tier this is fine. Revisit with prompt caching when we hit cost concerns.
- **MEDIUM nits**: orderBy enrichedAt for backfill (not publishedAt), cluster lead reaper, `(hours)::interval` → `make_interval`, `/api/cron/normalize` catch-up doesn't enrich.

## M3 plan (not started)

**Goal**: wire up `👍/👎/⭐` feedback persistence + basic editor auth.

### Schema additions
```sql
-- users: editors who provide feedback
CREATE TABLE users (
  id UUID PRIMARY KEY,              -- mirrors supabase auth.users.id
  email TEXT NOT NULL UNIQUE,
  role TEXT NOT NULL DEFAULT 'editor',  -- editor | admin | reader
  created_at TIMESTAMPTZ DEFAULT now()
);

-- feedback: one row per editor action
CREATE TABLE feedback (
  id SERIAL PRIMARY KEY,
  user_id UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  item_id INTEGER NOT NULL REFERENCES items(id) ON DELETE CASCADE,
  verdict TEXT NOT NULL,              -- up | down | save
  note TEXT,
  policy_version TEXT NOT NULL,       -- snapshot of policy at time of feedback
  created_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  UNIQUE (user_id, item_id, verdict)  -- idempotent
);
CREATE INDEX feedback_item_idx ON feedback (item_id, created_at);
CREATE INDEX feedback_recent_idx ON feedback (created_at DESC);
```

### Auth via Supabase
- Use `@supabase/ssr` + Supabase Auth magic-link email
- `app/api/auth/callback/route.ts` — handle session exchange
- Middleware extension — gate admin routes on valid session + role='editor'|'admin'
- Cookie-based session (ssr-safe, no client-side token handling)

### API
- `POST /api/feedback` — `{ itemId, verdict, note? }` → auth required, upsert by unique constraint
- `DELETE /api/feedback/:id` — undo
- `GET /api/admin/feedback?since=...` — paginated, admin-only

### UI wiring
- `components/feed/feedback-controls.tsx` — already has optimistic state; add real POST + revert on failure
- `app/[locale]/admin/iterations/page.tsx` — replace mock with real query
  - Metrics row: real counts from feedback table

  - Keep console/diff/version mock for M3 — those are M4 territory
