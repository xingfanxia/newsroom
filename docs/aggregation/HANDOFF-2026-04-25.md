## Critical learnings (do NOT lose)

1. **Drizzle array binding is a tuple, not an ARRAY.** Any time a JS array crosses into raw `sql\`\``, build the array with `sql.join`. See memory file `feedback_drizzle_array_binding.md`.
2. **Azure Foundry project URL ≠ Pro URL ≠ legacy URL.** Three different endpoint shapes. The Foundry one rejects top-level `system`. Keep the workaround until `@ai-sdk/openai` ships a fix.
3. **Defensive filters need code-path traces, not just intent.** Stage A's verified-filter was added "to respect Stage B's lock" but Stage A doesn't have a path that would violate the lock anyway. Filter just hid recall. Pattern: write down the harm prevented in code-path terms before adding.
4. **Hot feeds: time-first, importance-tiebreaker.** Importance-first surfaces old still-developing P1=100 events on top. The today filter admits them; the sort decides display order. Don't conflate.
5. **HNSW recall was NOT the problem.** ef_search=40, 100, 200 all returned the same nearest in our diagnostic queries. The handoff hypothesis was a red herring; root cause was the verified-filter (item 1 above).
6. **Cron stays on `*/30 * * * *`.** First production tick on gpt-5.5 ran fine: 11 arbitrations, 0 errors, 8 splits + 3 keeps. The Foundry workaround held under real cluster prompts.
7. **`as unknown as Array<T>`** remains the right cast for `client.execute(sql\`...\`)` — never `(result as { rows?: unknown[] }).rows?.[0]`. (Carried from previous session, still true.)
